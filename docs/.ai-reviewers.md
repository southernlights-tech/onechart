# AI Reviewer Guidelines for Southern Lights Infrastructure

When reviewing PRs in this repository, apply our company values and focus on infrastructure-specific concerns.

## About Southern Lights

Southern Lights is a DevOps consultancy specializing in Kubernetes orchestration, cloud-native solutions, and infrastructure automation. We help organizations modernize their infrastructure and operations through customized solutions that optimize efficiency and scalability. Our mission is to illuminate the path to digital transformation, enabling businesses to not just adapt to the future, but personalize it to their advantage.

Our expertise spans CI/CD, Infrastructure as Code, Cloud Management, Kubernetes adoption, Observability, and DevSecOps - always with a focus on client-specific needs and measurable results.

## Company Values in Code Review

### Integrity
- Are infrastructure changes accurately documented?
- Are limitations or tradeoffs transparently stated?
- Is the blast radius clearly communicated?
- Are post-mortems blameless and focused on systemic improvements?
- Is documentation grounded in facts, not aspirations?

### Courage
- Does this change address root cause or just symptoms?
- Are difficult architectural decisions acknowledged and justified?
- Are security concerns openly discussed rather than hidden?
- Does the PR admit mistakes or limitations honestly?

### Teamwork
- Does this enable other teams (platform engineering approach)?
- Are shared responsibilities in infrastructure clear?
- Will this create silos or foster collaboration?
- Are changes coordinated with affected teams?
- Does documentation help others understand and maintain this?

### Mastery
- Are decisions based on technical merit and CNCF best practices?
- Is the IaC idempotent and follows GitOps principles?
- Are Terraform/Kubernetes resources properly structured?
- Is there evidence of research and testing before implementation?
- Are technical reasons provided, not just hope or assumptions?

### Innovation
- Does this leverage cloud-native patterns appropriately?
- Are we adopting battle-tested CNCF solutions?
- Is there experimentation with new approaches in safe-to-fail environments?
- Does this improve upon existing patterns?

### Initiative
- Does this proactively solve a problem before it becomes critical?
- Are improvements proposed beyond the immediate requirement?
- Is automation being introduced to reduce toil?

### Continuous Improvement
- Are lessons from past incidents incorporated?
- Is observability built-in (metrics, logs, traces)?
- Are there clear feedback loops for measuring success?
- Is this change measurable and improvable?

## Infrastructure-Specific Review Checklist

### Security (DevSecOps)

**IAM & RBAC:**
- [ ] IAM policies follow least privilege principle
- [ ] Service accounts have minimal necessary permissions
- [ ] RBAC roles are scoped appropriately (namespace vs cluster)
- [ ] No overly permissive wildcards (*) in policies
- [ ] Cross-account access is properly assumed with conditions

**Secrets Management:**
- [ ] No secrets hardcoded in code or manifests
- [ ] Secrets are managed via Vault or SOPS
- [ ] Secret rotation strategy is documented
- [ ] Secrets are scoped to minimum necessary access
- [ ] External Secrets Operator is used for K8s secrets

**Network Security:**
- [ ] Network policies properly segment workloads
- [ ] Ingress/Egress rules are explicitly defined
- [ ] TLS is enforced for all external communication
- [ ] Private subnets are used where appropriate
- [ ] Security groups follow least privilege

**Compliance & Hardening:**
- [ ] Security contexts are defined for K8s workloads (runAsNonRoot, readOnlyRootFilesystem)
- [ ] Pod Security Standards/Policies are applied
- [ ] CIS benchmarks are followed (where applicable)
- [ ] Compliance requirements (SOC2/ISO27001) are met
- [ ] Vulnerability scanning is in place for images

### Infrastructure as Code (Terraform/NixOS)

**Terraform Best Practices:**
- [ ] Resources have appropriate tags/labels for organization and cost tracking
- [ ] State is properly managed (remote backend, locking enabled)
- [ ] Destroy operations are safe (prevent_destroy on critical resources)
- [ ] Module versions are pinned (avoid "latest" or branches)
- [ ] Variables have proper descriptions and types
- [ ] Outputs are documented and meaningful
- [ ] `terraform fmt` has been run
- [ ] `terraform validate` passes
- [ ] Changes reviewed with `terraform plan` output

**NixOS Configuration:**
- [ ] Configurations are reproducible and declarative
- [ ] No imperative system modifications
- [ ] Secrets are managed properly (not in Nix store)
- [ ] Service definitions follow NixOS conventions
- [ ] System updates are tested before production deployment

**General IaC:**
- [ ] Code is modular and reusable
- [ ] DRY principle is followed (avoid duplication)
- [ ] Comments explain "why" not "what"
- [ ] Breaking changes are clearly documented
- [ ] Migration path is provided for destructive changes

### GitOps & Kubernetes

**Flux CD:**
- [ ] Kustomizations are valid (`kustomize build` passes)
- [ ] HelmReleases use specific chart versions (not "latest")
- [ ] Flux reconciliation intervals are appropriate
- [ ] Dependencies between resources are explicit
- [ ] Namespace isolation is maintained
- [ ] Image update automation is configured correctly (if used)

**Kubernetes Resources:**
- [ ] Resource requests and limits are defined for all containers
- [ ] Health checks (liveness/readiness/startup probes) are configured
- [ ] Pod disruption budgets exist for HA workloads
- [ ] Affinity/anti-affinity rules are used for availability
- [ ] Priority classes are assigned appropriately
- [ ] ConfigMaps/Secrets are versioned to trigger rolling updates
- [ ] ServiceAccounts are explicitly defined (not default)

**Workload Configuration:**
- [ ] Images use specific tags (not "latest")
- [ ] Images are pulled from trusted registries
- [ ] Resource quotas are defined for namespaces
- [ ] Horizontal Pod Autoscalers are configured for scalable workloads
- [ ] Persistent volumes have appropriate storage classes and backup policies

### Observability (OpenObserve/OpenTelemetry)

**Logging:**
- [ ] Applications emit structured logs (JSON preferred)
- [ ] Log levels are appropriate (debug/info/warn/error)
- [ ] Sensitive data is not logged
- [ ] Logs are forwarded via Fluent Bit
- [ ] Log retention policies are defined

**Metrics:**
- [ ] Key metrics are exposed (latency, errors, saturation, traffic)
- [ ] Metrics follow naming conventions (Prometheus format)
- [ ] OpenTelemetry instrumentation is configured
- [ ] Custom metrics are documented
- [ ] Metric cardinality is reasonable (avoid label explosion)

**Tracing:**
- [ ] Distributed tracing is enabled for critical paths
- [ ] Trace sampling rates are configured appropriately
- [ ] Trace context is propagated across service boundaries
- [ ] Spans include relevant attributes

**Alerting:**
- [ ] Alerts are defined for critical failure modes
- [ ] Alert thresholds are based on SLIs/SLOs
- [ ] Alerts are actionable (clear remediation steps)
- [ ] Alert fatigue is avoided (proper tuning)
- [ ] Runbooks are linked from alerts

**Dashboards:**
- [ ] Dashboards exist in OpenObserve for key services
- [ ] Dashboards show RED metrics (Rate, Errors, Duration)
- [ ] Dashboards are organized and labeled clearly

### Resilience & Disaster Recovery

**High Availability:**
- [ ] Multi-AZ deployment for critical services
- [ ] Failover mechanisms are in place and tested
- [ ] Load balancing distributes traffic appropriately
- [ ] Circuit breakers prevent cascading failures
- [ ] Graceful degradation is implemented

**Backup & Recovery:**
- [ ] Backup strategy is documented and automated
- [ ] Backup retention follows compliance requirements
- [ ] Recovery procedures are documented (runbooks)
- [ ] Recovery Time Objective (RTO) and Recovery Point Objective (RPO) are defined
- [ ] Backups are tested regularly (restore tests)

**Blast Radius Limitation:**
- [ ] Changes are deployed incrementally (canary/blue-green)
- [ ] Resource isolation prevents total outages
- [ ] Rate limiting protects critical services
- [ ] Timeouts prevent resource exhaustion

### CI/CD & Automation

**GitHub Actions:**
- [ ] Workflows validate changes before merge (terraform plan, kustomize build)
- [ ] Security scanning is integrated (Gitleaks, Semgrep, Trivy, Checkov)
- [ ] Tests run automatically on PRs
- [ ] Workflows use pinned action versions (SHA or major version)
- [ ] Secrets are stored in GitHub Secrets, not hardcoded
- [ ] Workflow runs are idempotent

**Pipeline Security:**
- [ ] Least privilege for CI/CD service accounts
- [ ] Branch protection rules are enforced
- [ ] Required checks must pass before merge
- [ ] Code review required from CODEOWNERS
- [ ] No force pushes to protected branches

### Cost Optimization

- [ ] Right-sized resources (not over-provisioned)
- [ ] Autoscaling configured to scale down during low usage
- [ ] Spot/preemptible instances used where appropriate
- [ ] Unused resources are identified and cleaned up
- [ ] Cost allocation tags are applied
- [ ] Reserved instances/savings plans considered for steady-state workloads

### Documentation

- [ ] README or runbook explains the change
- [ ] Architecture decisions are documented (ADRs if applicable)
- [ ] Dependencies and prerequisites are clear
- [ ] Disaster recovery procedures are updated
- [ ] Terraform/K8s resources have meaningful descriptions
- [ ] Breaking changes are highlighted in PR description
- [ ] Migration steps are provided for complex changes

## Common Anti-Patterns to Flag

### Security Anti-Patterns
- ❌ Hardcoded credentials, API keys, or certificates
- ❌ Secrets in environment variables without encryption
- ❌ Running containers as root unnecessarily
- ❌ Overly permissive IAM policies (e.g., `*` actions on `*` resources)
- ❌ Missing TLS/SSL for data in transit
- ❌ Publicly accessible S3 buckets or databases without justification

### Infrastructure Anti-Patterns
- ❌ Mutable infrastructure (manual changes to servers)
- ❌ Unmanaged Terraform state or missing state locking
- ❌ Using "latest" tags for images or chart versions
- ❌ No resource limits leading to noisy neighbor problems
- ❌ Single points of failure (no redundancy)
- ❌ Configuration drift between environments

### Operational Anti-Patterns
- ❌ No monitoring or alerting for new services
- ❌ Missing health checks causing failed deployments
- ❌ No disaster recovery plan or untested backups
- ❌ Lack of documentation for complex systems
- ❌ No rollback plan for risky changes
- ❌ Changes deployed directly to production without staging

### Code Quality Anti-Patterns
- ❌ Duplicate code across multiple Terraform modules
- ❌ Hardcoded values that should be variables
- ❌ No validation for input variables
- ❌ Unclear or missing comments for complex logic
- ❌ Inconsistent naming conventions
- ❌ No testing of infrastructure code changes

## CNCF & Cloud-Native Best Practices

When reviewing infrastructure changes, ensure alignment with CNCF principles:

**Cloud-Native Patterns:**
- Prefer graduated or incubating CNCF projects over proprietary solutions
- Design for failure and resilience
- Use declarative configuration (GitOps)
- Implement observability from the start
- Automate operations (reduce manual toil)
- Design for portability (avoid vendor lock-in where possible)

**12-Factor App Principles (for services we run):**
- Configuration via environment variables (externalized config)
- Treat backing services as attached resources
- Stateless processes (state in external stores)
- Disposability (fast startup and graceful shutdown)
- Logs as event streams
- Dev/prod parity

**Reference CNCF Tools (when applicable):**
- **Container Orchestration:** Kubernetes (EKS, K3s)
- **GitOps:** Flux CD
- **Service Mesh:** (if adopted: Istio, Linkerd)
- **Observability:** OpenTelemetry, Fluent Bit, OpenObserve
- **Security:** Vault (secrets), Falco (runtime security), Trivy (vulnerability scanning)
- **Policy:** OPA/Gatekeeper (admission control)

## Review Workflow

1. **Understand the Context:** Read the PR description and linked issues/tickets
2. **Apply Company Values:** Assess against Integrity, Courage, Teamwork, Mastery, Innovation, Initiative, Continuous Improvement
3. **Check Security First:** Security issues block merge
4. **Validate Infrastructure:** Ensure IaC follows best practices
5. **Assess Observability:** Confirm monitoring and alerting are in place
6. **Review Resilience:** Check disaster recovery and blast radius
7. **Verify Documentation:** Ensure changes are documented for future maintainers
8. **Flag Anti-Patterns:** Call out common mistakes
9. **Provide Constructive Feedback:** Explain "why" behind suggestions
10. **Approve or Request Changes:** Based on holistic assessment

## Example Good Review Comment

> **Security Concern:** This IAM policy grants `s3:*` on all buckets. Per our principle of **Mastery** (least privilege), let's scope this to only the specific buckets needed: `arn:aws:s3:::our-app-bucket/*`. This reduces blast radius if credentials are compromised.
> 
> **Observability Gap:** Missing Prometheus metrics for this new service. Per **Continuous Improvement**, let's add RED metrics (request rate, error rate, duration) so we can detect issues in production.
> 
> **Great Job:** Love that you included a runbook in the PR description. This demonstrates **Teamwork** by making it easy for anyone on-call to handle issues. 💪

## Example Constructive Feedback

> **Infrastructure Drift Risk:** I notice manual `kubectl apply` commands in the deployment steps. Per our **GitOps** principle and **Integrity** value (reproducibility), let's commit these manifests to the Flux repo instead so changes are versioned and auditable.
>
> **Suggestion:** Add a Kustomization in `infrastructure/apps/prod/` that references this service. Happy to pair on this if helpful!

---

**Remember:** The goal is not to block progress, but to build secure, reliable, observable infrastructure that reflects our company values. When in doubt, favor transparency, security, and collaboration.
